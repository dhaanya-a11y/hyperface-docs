---
title: Milestone Offer
---

<Info>
A **Milestone Offer** rewards customers for achieving a **predefined spending or transaction milestone** within a configured subscription cycle. Unlike Transactional Offers, customer progress is accumulated across multiple qualifying transactions until the configured milestone threshold is reached.
</Info>

### Key characteristics

- **Cumulative progress tracking** – Eligible transactions accumulate toward the configured milestone threshold until it is achieved.
- **Milestone-based qualification** – Customers become eligible for rewards only after completing the configured spending or transaction milestone.
- **Configurable outcome** – Rewards can be issued as Cashback, Reward Points, Voucher, Smart Tag, Accelerated Reward, or Partner Points.
- **Subscription cycle support** – Milestones are evaluated within the configured subscription cycle and can repeat based on the configured number of cycles.

The **Offer Type** is the foundation of the offer configuration journey. It determines how customers qualify for milestone-based rewards.

For this guide, select **Milestone Offer**, which is designed to reward customers after they successfully complete predefined spending or transaction milestones. This offer type is commonly used for loyalty campaigns, spending challenges, milestone rewards, and recurring engagement programs.

## Configure the Offer Scope

![Step 4 - Configure Basic Offer Details](/images/0.4.png)

The **Offer Scope** defines where the Milestone Offer will be applicable. In this step, configure the basic offer details and specify the issuer and program(s) for which the offer should be available.

Configure the following:

- **Offer Name** – Enter a unique name for the offer.
- **Offer Description** – Provide a brief description of the offer.
- **Offer Duration** – Select the offer validity period.
- **Eligibility Level** – Choose whether the offer should be evaluated at the **Account Level** or **Customer Level**.

![Step 4 - Configure Offer Applicability](/images/0.4.1.png)

Define the applicability of the offer by configuring:

- **Issuer** – Select the issuer.
- **Program Type** – Select the applicable product category.
- **Apply to all current and future programs** – Enable this option to apply the offer across all existing and future programs.
- **Program** – Select specific program(s) if the offer should not apply to all programs.
- **Enable Progress Transfer on Variant Upgrade** – Preserve a customer's milestone progress when they upgrade to another supported card or account variant.

<Info>
Without this enabled, a customer who upgrades mid-cycle starts their milestone progress over on the new variant. Enable it for campaigns where you don't want an upgrade to reset a customer's accumulated progress.
</Info>

### Example

Create a Milestone Offer by entering the offer details, selecting the required eligibility level, and choosing the applicable issuer and program(s). Enable **Apply to all current and future programs** if the offer should automatically apply to every program under the selected program type.

➡️ **Continue to:** **[Offer Scope](/offer/offer-scope)**

## Configure Offer User Base

![Step 5 - Select Static Base](/images/0.5.png)

Configure the **Offer User Base** to define which users are eligible for the Milestone Offer.

First, select the type of user base:

- **Static Base** – Uses a fixed list of eligible users that remains unchanged after the offer goes live.
- **Dynamic Config** – Continuously evaluates users against the configured eligibility criteria, allowing new users to become eligible over time.

Next, define the eligibility criteria for the selected user base.

Configure the following:

- **Eligibility** – Select the required user group or account scope.
- **Add Criteria** – Add one or more conditions to further refine the eligible user base.
- **Estimate User Base** – Calculate the estimated number of users that satisfy the configured criteria before publishing the offer.

### Example

Suppose the offer should be available only to customers enrolled through a Smart Tag or a defined user segment. Select the appropriate user base type, configure the eligibility criteria, add additional filters if required, and use **Estimate User Base** to verify the expected audience before proceeding.

<Tip>
Choose **Static Base** when the eligible audience is predefined and unlikely to change. Choose **Dynamic Config** when new customers should keep joining the eligible base as they meet the criteria — useful for always-on milestone campaigns rather than one-time cohorts.
</Tip>

➡️ **Continue to:** **[Offer User Base](/offer/offer-user-base)**

## Configure Milestone Eligibility

![Step 6 - Configure Milestone Eligibility](/images/Screenshot%202026-07-17%20105521.png)

Configure the milestone conditions that determine when a customer becomes eligible for the configured reward.

<Info>
**Offer User Base** determines *who* is evaluated for this offer. **Milestone Eligibility** and **Transaction Eligibility** — configured below — determine *which of their transactions* actually count toward the milestone. A customer can belong to the eligible user base and still never complete the milestone if their transactions don't satisfy these conditions.
</Info>

First, define how milestone progress should be tracked by selecting the appropriate aggregation method and configuring the milestone threshold.

Configure the following:

- **Aggregate By** – Select whether the milestone should be calculated based on the **Sum of Transaction Values** or the **Count of Transactions**.
- **Threshold** – Configure the comparison operator and the milestone value that customers must achieve.
- **Lock Milestone Progress on Achievement** – Enable this option to stop tracking progress after the milestone has been achieved.
- **Subscription Cycle Period** – Select the duration over which milestone progress should be tracked.
- **Number of Cycles** – Specify how many times the milestone can be repeated during the offer period.

<Warning>
Changing **Lock Milestone Progress on Achievement** or the **Subscription Cycle Period** after the offer is active may affect how in-progress customer milestones are evaluated. Review these settings carefully before publishing.
</Warning>

➡️ **Continue to:** **[Milestone Eligibility](/offer/milestone-eligibility)**

### Transaction Eligibility

![Step 6 - Configure Transaction Eligibility](/images/Screenshot%202026-07-17%20105533.png)

Configure the transaction conditions that determine which transactions contribute towards milestone completion.

Select the required **Transaction Attribute**, choose the appropriate **comparison operator**, and provide the corresponding value. Add additional criteria if required and select the **Processing Date** used for milestone evaluation.

### Supported Transaction Attributes

- MCC
- MID
- TID
- POS Entry Mode
- Transaction Value
- Transaction Type
- Merchant Name
- Payer VPA
- Payee VPA

### Example

Suppose customers should receive a reward after completing **5 eligible transactions** of **₹500 or more** during a monthly subscription cycle. Configure the aggregation as **Count of Transactions**, set the threshold to **5**, and add transaction eligibility conditions to include only qualifying transactions.

<Tip>
Set **Number of Cycles** higher than 1 to turn a one-time milestone into a repeating reward — for example, rewarding customers every month they hit the spend threshold, rather than only once for the life of the offer.
</Tip>

## Configure Offer Calculation

![Step 7 - Configure Offer Calculation](/images/0.6.png)

Configure how the reward should be calculated once a customer successfully completes the configured milestone.

Select the appropriate **Offer Calculation Rule** based on the selected outcome type and configure the reward value that should be awarded after milestone completion.

Configure the following:

- **Offer Calculation Rule** – Select the calculation method for the offer outcome.
- **Outcome Value** – Specify the reward amount or value to be awarded after the milestone is achieved.
- **Skip Reward Capping** – Enable or disable reward capping at the required level, such as **Program**, **Transaction**, **MCC**, **MID**, or **Currency**.
- **Allow Reversal** – Enable this option to reverse the reward if one or more qualifying transactions are subsequently reversed.

<Info>
Reward capping limits the maximum payout per customer at the configured level, independent of the milestone threshold itself. A customer can hit the spending milestone and still receive a capped reward if the calculated value exceeds the configured cap.
</Info>

### Example

Select the required calculation rule, configure the reward value, choose the applicable reward capping settings, and enable **Allow Reversal** if rewards should be reversed when qualifying transactions are reversed.

➡️ **Continue to:** **[Offer Calculation](/offer/offer-calculation)**

## Configure Posting

![Step 9 - Configure Posting](/images/l.png)

Configure when and how the reward should be posted to eligible customers after they successfully complete the configured milestone.

Configure the following:

- **Enable Posting** – Enable or disable reward posting.
- **Posting Eligibility** – Define the conditions that must be satisfied before the reward is posted.
- **Offer Posting** – Choose whether the reward should be posted immediately after milestone completion or scheduled for a later time.
- **Reward Point Expiry** – Configure the validity period for reward points, if applicable.
- **Narration** – Enter the ledger narration for both reward posting and reward reversal.

![Step 9 - Configure Narration](/images/l.1.png)

### Example

Enable posting, configure the required posting eligibility, choose the appropriate posting method, specify the reward point expiry if applicable, and provide the posting and reversal narration.

<Warning>
Changing the posting schedule after an offer is active may affect when future rewards are credited. Review the configured posting option before publishing the offer.
</Warning>

![Step 10 - Configure Offer Details Page](/images/n.png)
