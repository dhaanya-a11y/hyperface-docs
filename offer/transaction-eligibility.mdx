---
title: Transaction Eligibility
description: Configure the transaction conditions that determine whether a transaction qualifies for an offer.
---

Defines the conditions a transaction must satisfy for an offer to apply to it. Once a customer or account qualifies under the **Offer User Base**, every transaction they make is checked against the conditions configured here. Only transactions that satisfy these conditions are treated as qualifying transactions and move forward to **Offer Calculation**.

Transaction Eligibility is configured using **Transaction Attributes**—properties of a transaction such as the merchant, terminal, amount, or payment channel used.

<Note>
**Offer User Base** determines *who* is evaluated for the offer. **Transaction Eligibility** determines *which of their transactions* qualify. A customer belonging to the eligible user base still needs to make a transaction that satisfies the configured Transaction Eligibility conditions before any reward is calculated.
</Note>

## Transaction Attributes

Select one or more transaction attributes to define the conditions a qualifying transaction must satisfy.

| Transaction Attribute | What It Evaluates | Example |
|------------------------|--------------------|---------|
| **MCC** | The Merchant Category Code (MCC) recorded against the transaction. | 5541 (Fuel Stations) |
| **MID** | The Merchant Identifier (MID) of the merchant where the transaction was made. | A specific merchant's MID |
| **TID** | The Terminal Identifier (TID) of the terminal used for the transaction. | A specific POS terminal's TID |
| **POS Entry Mode** | The mode used to capture card details at the point of sale. | Contactless |
| **Transaction Value** | The amount of the transaction. | Greater than ₹1,000 |
| **Transaction Type** | The type of transaction performed. | Purchase |
| **Sovereignty Indicator** | Whether the transaction was made domestically or internationally. | Domestic |
| **Transaction Code** | The transaction code associated with the transaction. | A specific transaction code |
| **Payer VPA** | The Virtual Payment Address (VPA) of the customer making a UPI payment. | customer@upi |
| **Payee VPA** | The Virtual Payment Address (VPA) of the recipient of a UPI payment. | merchant@upi |

<Tip>

Not every attribute applies to every transaction.

- **Payer VPA** and **Payee VPA** are applicable only to UPI transactions.
- **MCC**, **MID**, and **TID** are applicable only to card transactions.

Select the transaction attributes that match the payment channel and transaction type your offer is intended for.

</Tip>

## Configure Transaction Conditions

<Warning>
All configured transaction conditions are evaluated together. Ensure the selected transaction attributes, comparison operators, and values accurately reflect the intended eligibility criteria before proceeding.
</Warning>

For each transaction attribute selected:

- Select the appropriate comparison operator.
- Enter or select the value the attribute should be compared against.

![Step 6 - Configure Transaction Conditions](/images/eligibility-conditions.png)

### Add Multiple Conditions

Add more than one transaction condition if the offer should evaluate more than one attribute—for example, requiring both a specific **MCC** and a minimum **Transaction Value**. All configured conditions apply to the same transaction eligibility check.

### Transaction Posting Date

Select the **Transaction Posting Date** to specify the date used when checking whether a transaction falls within the offer's evaluation window.

## Example: Cashback on High-Value UPI Transactions

To reward customers for UPI payments above ₹500:

1. In **Offer User Base**, configure the eligible customer base.
2. In **Transaction Eligibility**, select **Transaction Type** and set it to `UPI`.
3. Select **Transaction Value**, set the comparison operator to `Greater than`, and enter `500`.
4. Select the **Transaction Posting Date** for evaluation.
5. Click **Continue** to proceed to **Offer Calculation**.

Only UPI transactions above ₹500 made by customers in the configured user base are treated as qualifying transactions.

## What Happens After Configuration?

Every transaction made by a customer in the eligible user base is checked independently against the configured Transaction Eligibility conditions. A transaction that satisfies all configured conditions becomes a qualifying transaction and proceeds to **Offer Calculation**, where the reward is computed. There is no requirement to accumulate spend or transaction count—each transaction is evaluated on its own.

## Complete the Configuration

After configuring the required transaction conditions:

1. Review the selected transaction attributes, operators, and values.
2. Verify the configured **Transaction Posting Date**.
3. Click **Continue** to proceed to the **Offer Calculation** configuration.
