## Introduction

This guide explains how to create and configure Transactional Offers in the Hyperface Dashboard.

Transactional Offers enable issuers to reward customers for performing qualifying individual transactions. Eligibility is configured using transaction-level conditions such as MCC, MID, Transaction Amount, POS Entry Mode, and other transaction attributes.

This document covers the complete offer creation journey, offer listing management, and key field-level guidance.

---

## What are Transactional Offers?
Transactional Offers are reward campaigns triggered when a customer completes a single qualifying transaction.

Unlike Milestone Offers, no spend accumulation is required. Reward eligibility is evaluated individually for each transaction.

### Key Characteristics

Transactional Offers are evaluated at the transaction level rather than at an aggregate spend level.

Key characteristics include:

* Rewards are triggered by individual qualifying transactions.
* Eligibility is evaluated in real time or near real time based on configured rules.
* Multiple transaction attributes can be combined to define eligibility.
* Rewards can be issued as cashback, reward points, vouchers, or Smart Tags.
* Usage limits can be configured to control customer earning frequency.
* Posting rules determine when rewards become visible to customers.

Transactional Offers are ideal for merchant promotions, spend-based rewards, customer activation campaigns, and targeted engagement initiatives.

## Create a Transactional Offer

Follow the steps below to create a new Transactional Offer.

1. Navigate to the Offer Listing page.
2. Select the appropriate Offer Category tab.

4. Select **Transactional Offer**.

## Step 1: Configure Offer Details
### Important Notes

- Offer Name does not need to be unique.
- Offer Description is for internal reference only.
- Start time must be at least 5 minutes in the future.
- End date must be later than the start date.
- Offers are created in Scheduled status and automatically become Active at the configured start time.
- If "Apply to all programs" is enabled, individual program selection is disabled.

## Step 2: Configure Offer User Base
### User Base Selection Guidelines

Selecting the correct user base is important because it determines who can participate in the offer.

Use Static Base when the eligible customer population is known in advance and should remain fixed throughout the offer period.

Use Dynamic Configuration when customers should be able to become eligible after the offer has gone live.

### Example Scenarios

#### New Customer Campaign

Target customers who opened an account within the last 30 days.

Configuration:

* User Base: Static Base
* Smart Tag: New Customers
* Condition: Allowed

#### Opt-In Cashback Campaign

Allow customers to enroll themselves during the campaign period.

Configuration:

* User Base: Dynamic Configuration
* Enrollment Type: Opt-In

### Best Practices

* Validate Smart Tag membership before publishing.
* Review inclusion and exclusion criteria carefully.
* Use Dynamic Configuration only when ongoing enrollment is required.
* Test enrollment scenarios before activation.


Define which customers are eligible for the offer.

### Static Base

* Smart Tag Filtering
* Profile Attribute Filtering

### Dynamic Configuration

* Opt-In Enrollment
* Smart Tag Attachment

### Smart Tag Options

* Allowed

## Step 3: Configure Transaction Eligibility
When configuring transaction eligibility:

* Use meaningful list names for MCCs, MIDs, and merchant groups.
* Verify all condition logic before publishing.
* Minimize unnecessary eligibility conditions to reduce maintenance complexity.
* Test common transaction scenarios before activating the offer.
* Review Denied conditions carefully to ensure valid transactions are not unintentionally excluded.

### Example Eligibility Scenario

Objective:

Reward customers for fuel purchases above ₹1,000.

Configuration:

* MCC = Fuel MCC List
* Transaction Amount > ₹1,000

Result:

Only transactions matching both conditions will qualify for reward evaluation.

### Eligibility Logic

All conditions within a group use AND logic.

A customer becomes eligible when a transaction satisfies all configured conditions within a group.

Conditions can be configured using:

- Allowed
- Denied
- List Based Inputs
- Direct Input Values

### Creating Lists

Lists can be created directly from the eligibility screen.

Examples include:

- Shopping MCCs
- Fuel MCCs
- Merchant MID Lists

Lists can be configured as Immutable or Editable depending on business requirements.

## Step 4: Configure Offer Calculation
### Calculation Configuration Guidelines

The reward calculation determines the value customers receive after completing a qualifying transaction.

The selected calculation method should align with campaign objectives, reward budgets, and expected customer behaviour.

### Choosing the Right Calculation Type

#### Fixed Reward

Best suited for simple campaigns where every qualifying transaction should earn the same reward.

#### Percentage Based Reward

Useful when rewards should scale based on transaction value.

#### Slab Based Reward

Recommended when higher transaction values should earn progressively larger rewards.

### Validation Checklist

Before publishing:

* Verify reward amounts.
* Validate percentage calculations.
* Review slab ranges for overlaps.
* Confirm reward caps where applicable.
* Test sample transaction scenarios.


### Calculation Examples

#### Fixed Reward

Every qualifying transaction earns a predefined reward amount.

Example:

- Qualifying supermarket transaction → 50 reward points

#### Percentage Based Reward

Reward is calculated as a percentage of transaction value.

Example:

- Spend ₹2,000
- Reward = 5%
- Earn 100 reward points

#### Slab Based Reward

Rewards vary based on transaction value ranges.

Example:

- ₹500–₹1,999 → 50 points
-- ₹2,000–₹4,999 → 150 points

## Step 6: Configure Posting
### Posting Best Practices

Posting configuration determines when rewards become visible to customers.

Selecting the appropriate posting method helps balance customer experience with operational risk.

### Immediate Posting

Recommended when:

* Reversal risk is minimal.
* Customer gratification is a priority.
* Rewards should appear as quickly as possible.

### Scheduled Posting

Recommended when:

* Transactions may be refunded or reversed.
* Campaigns require validation periods.
* Additional fraud or compliance checks are necessary.

### Operational Recommendations

* Configure posting delays based on expected reversal windows.
* Use meaningful posting narrations for audit purposes.
* Verify reversal narrations before publishing.
* Test reward posting behaviour before activating the campaign.

### Posting Recommendations

Scheduled Posting is recommended because it provides time for transaction reversals and refunds before rewards are issued.

Immediate Posting should only be used when reversal risk is minimal.

- Posting Narration
- Reversal Narration

## Step 8: Review and Publish

Review all configurations before publishing

Before publishing an offer, verify:

* Offer details are accurate.
* Program selection is correct.
* User base configuration has been validated.
* Eligibility rules have been reviewed.
* Reward calculations have been tested.
* Usage limits align with campaign objectives.
* Posting configuration is complete.
* Customer-facing display content has been approved.

Publishing should only be performed after all configurations have been reviewed and validated.


### Review Checklist

* Offer Details
* User Base Configuration
* Transaction Eligibility
* Offer Calculation
* Usage Limits
* Posting Configuration
* Display Elements

Click **Publish** once all configurations have been validated.

The offer status becomes **Scheduled** and automatically changes to **Active** at the configured start time.

## Managing Existing Offers

The Offer Listing page allows users to manage existing offers throughout their lifecycle.

### Available Actions

- View
- Edit
- Duplicate
- Delete
- Expire

### Offer Statuses

#### Active

Offer is currently live and available to eligible customers.

#### Scheduled

Offer has been configured and will become active at a future date.

#### Draft

Offer has been created but not yet activated.

#### Expired

Offer has reached its end date and is no longer active.

### Viewing Offer Details

Click any offer name from the Offer Listing page to view:

- Eligibility Rules
- Calculation Rules
- Usage Limits
- Posting Configuration
- Display Configuration

### Important Notes

- Active offers can only be edited for display settings and end date.
- Eligibility and calculation rules are locked once the offer becomes active.
- Duplicated offers are created in Draft status.
- Scheduled offers can be deleted before activation.
### Viewing Offer Details

Click any offer name from the Offer Listing page to view the complete offer configuration, eligibility rules, calculation settings, usage limits, posting setup, and display configuration.
