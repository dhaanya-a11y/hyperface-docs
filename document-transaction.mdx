Introduction
This guide explains how to create and configure Transactional Offers in the Hyperface Dashboard. Transactional Offers enable issuers to reward customers for performing qualifying individual transactions. Eligibility is configured using transaction-level conditions such as MCC, MID, Transaction Amount, POS Entry Mode, and other transaction attributes.

This document is intended for teams responsible for creating and managing offers within the Hyperface platform. It covers the complete offer creation journey, offer listing management, and key field-level guidance.

What are Transactional Offers?
Transactional Offers are reward campaigns triggered by a customer completing a single qualifying transaction. Unlike milestone offers, no spend accumulation is required - reward eligibility is evaluated per transaction.

A Transactional Offer consists of the following components:

Offer Details - Basic identity, type, duration, and scope of the offer.
User Base Configuration Defines which customers are eligible to opt in.
Transaction Eligibility Rules - Defines qualifying transaction conditions using AND logic.
Offer Calculation Rules - Defines how the reward is calculated.
Usage Limits - Defines per-customer earning frequency limits.
Posting Configuration - Defines when rewards are posted to customers.
Display Elements - Defines the visual elements of a benefit on the PWA

Once all configurations are completed and reviewed, the offer is published and becomes available to eligible customers from the configured start datetime.

Navigating to the Offer Listing Page
The Offer Listing page is the central hub for viewing and managing all offers configured in the system.

Navigate to the Offers section from the left navigation menu.
Select Manage Offers.
The Offer Listing page will appear.

Offer Category Tabs
Offers are grouped into categories displayed as tabs at the top of the listing page. Select the appropriate tab based on the offer type being configured. Navigate to the correct category tab before creating a new offer.

Summary Counters
Key metrics are displayed at the top of the listing page for a quick overview:

Active Now - Number of currently live and active offers.
Approval Pending - Offers awaiting checker approval.
Revision Required - Offers returned to the maker for corrections.
Expiring This Week - Offers whose end date falls within the current week.

Filter and Search
Use the Filter Bar to narrow down offers by Status, Scope, Outcome Type, Offer Type, Program, Created By, Start Date, or End Date. Use the Search Bar to find a specific offer by name or offer ID - the search uses partial match logic.


Create a Transactional Offer
Follow the steps below to create a new Transactional Offer. The creation journey is a sequential multi-step flow. Each step must be saved before proceeding to the next. You can navigate back to a completed step at any time using the left navigation panel.

On the Offer Listing page, navigate to the correct offer category tab.
Click the Create New Offer button at the top right of the listing page.
The offer creation journey will open with the left navigation panel showing all steps.
On the Offer Type selection screen, select Transactional Offer.
Click Continue to proceed to the Outcome Type screen.
Select the outcome type that the customer should receive on completing the offer (e.g. Cashback, Voucher, Reward Points).
Click Continue to proceed to the Offer Scope screen where you will enter the basic details.

Note:
The Offer Type and Outcome Type selections are made at the start of the journey and determine the configuration options available in subsequent steps.
Only offer types and outcome types that are relevant to your program scope will be applicable. 



Step 1: Configure Offer Details
This section captures the basic identity and scope of the offer. It includes the offer name, description, active duration, eligibility level, and the program(s) this offer applies to.

Enter the Offer Name.
Enter the Offer Description (for internal reference).
Configure the Offer Duration - set a Start Date and End Date with time.
Select the Issuer - Bank from the dropdown.
If the offer should apply to all current and future programs under the selected type, enable the 'Apply to all current and future programs' toggle. When enabled, the individual Program selection field will be disabled.
If the toggle is not enabled, select the specific Program(s) from the dropdown.
Verify all details and click Save & Continue.

Step 2: Configure Offer User Base
This step determines which customers can participate in the offer and how they are enrolled.

Select the User Base type:
Static Base - User list is locked at the time the offer goes live.
Dynamic Config - Allows new eligible users to enrol throughout the offer period. Select this for Opt-In based offers.
For Static Base:
Select Smart Tag option if offer is for a segmented customer cohort
Select the desired smart tag name and choose Allowed or Denied to include or exclude customers from the offer
Profile attributes such as birthdate and account creation date are also available for filtering cohorts.
For Dynamic Base:
Opt-In - Customers manually opt in through the customer-facing interface.
Smart Tag Attachment - Customers are auto-enrolled based on Smart Tag membership.
Verify the user base configuration and click Save & Continue.

Note:
To run the offer for all accounts in the given scope, select Static Base and do not configure any user base eligibility

Step 3: Configure Transaction Eligibility
This step defines the transaction-level conditions that must be satisfied for the offer to trigger. Conditions can be combined using AND and OR logic to model complex eligibility scenarios.

Each qualifying transaction that satisfies all configured conditions will trigger reward evaluation.

Adding Conditions (AND Logic)
Click Add Condition to add an eligibility filter within the current group.
Select the attribute to filter on from the dropdown. Available attributes include:
MCC (Merchant Category Code)
MID (Merchant ID)
TID (Terminal ID)
Merchant Name
Transaction Value
Transaction Type
POS Entry Mode
Sovereignty Indicator
Transaction Code
For the selected attribute, choose the input method - List (select from a pre-configured list) or Input (type values directly).
Select the Action:
Allowed - Transactions matching this attribute value are included.
Denied - Transactions matching this attribute value are excluded.
Click Add Condition again to add more conditions within the same group. All conditions within a group are joined by AND logic - a transaction must satisfy all of them.

Creating a New List
When selecting an attribute that accepts a List input (e.g. MCC or MID), you can create a new named list of values directly from within the eligibility screen. This is useful when the values do not already exist as a saved list.

In the attribute row, set the input method to List.
Click the list dropdown and select Create New List.
The Create New List form will open. Fill in the following fields:
Name - A unique name for the list (e.g. 'Shopping MCCs', 'Shopee MIDs'). Each list name must be unique.
Items - Enter values that are comma separated which must be part of the list OR
Choose File - Upload a CSV file containing the values for the list.
Issuer - Select the relevant issuer from the dropdown.
Immutable - Select True if the list values should be locked after creation. Select False if the list may need to be updated in future.
Click Submit to save the list.
The newly created list will now be available to select in the attribute row.

Note:
Items field and Client field in the Create New List form can be skipped.
If a list is set to Immutable = False, it can be updated later. Any updates to a list will apply to all active offers using that list - including in-progress customer evaluations.
Transaction Value conditions are also used to render progress bars in the customer-facing interface. Ensure the threshold values are configured accurately.


Once all conditions and groups are configured, verify the logic and click Save & Continue.


Note:
A customer becomes eligible when a transaction satisfies all conditions in at least one group.







Step 4: Configure Offer Calculation
This step defines how the reward is calculated when a qualifying transaction is identified.

The Offer Calculation screen will appear.
Select the reward type from the available options. Examples include:
Fixed Reward - A flat reward amount per qualifying transaction.
Percentage Based Reward - A percentage of the qualifying transaction amount.
Slab Based Reward - Tiered rewards based on transaction amount ranges.
Other reward structures available in the dropdown.
Enter the required values for the selected reward type.
Configure minimum or maximum reward caps if applicable.
Verify the reward calculation logic and click Save & Continue.

Note:
Reward calculation is applied per qualifying transaction unless otherwise configured.
Minimum and maximum reward caps are optional but recommended to control reward exposure.


Offer Calculation Examples
Fixed Reward
 Every qualifying transaction earns a flat reward, regardless of transaction amount.
 Example: Customer makes a transaction at a supermarket MCC → earns 50 reward points flat.
Percentage Based Reward
 Reward is a percentage of the transaction amount.
 Example: Customer spends 2,000 on a qualifying MCC → earns 5% = 100 reward points. Optionally cap the maximum reward at 500 points per transaction.
Slab Based Reward
 Reward tier depends on which spending bracket the transaction falls into.
 Example:
Transaction value 500–1,999 → earn 50 points
Transaction value 2,000–4,999 → earn 150 points




Step 5: Configure Usage Limits
This step controls how often an individual customer can earn rewards through this offer. All limit types are optional - configure only the limits relevant to your campaign design.

The Usage Limits screen will appear.
Configure the applicable limits:
Daily Limit - Maximum number of times a customer can earn per day.
Weekly Limit - Maximum earnings per week.
Monthly Limit - Maximum earnings per month.
Lifetime Limit - Maximum total earnings over the entire offer period.
Transaction Count Limit - Maximum number of qualifying transactions that will be rewarded.
Leave a limit field empty if no restriction is required for that time period.
Verify the configured limits and click Save & Continue.

Note:
Usage limits apply per individual customer.
If no limits are configured, a customer can earn rewards for every qualifying transaction within the offer period and budget.
This screen can be skipped if no usage limits are required for the offer.


Usage Limits Examples
Daily Limit
 A customer can earn a reward a maximum of 2 times per day, regardless of how many qualifying transactions they make.
Weekly Limit
 A customer can earn rewards on up to 5 qualifying transactions within a calendar week.
Transaction Count Limit
Only the first 2 qualifying transactions per customer will be rewarded. Subsequent qualifying transactions within the same period will not trigger a reward.

Combined Example
Daily limit = 2, Monthly limit = 10, Lifetime limit = 25. A customer who makes 5 qualifying transactions in a day will only earn rewards for the first 2. They can earn up to 10 in the month and no more than 25 over the full offer period.







Step 6: Configure Posting
This step defines when and how rewards are posted to customers after a qualifying transaction is identified and validated.

The Posting Configuration screen will appear.
If Hyperface is the system of records for rewards: Ensure the Enable Posting toggle is turned ON. This controls whether the reward is posted to the customer's account. The customer will only see the reward in their interface after posting is completed.
If Hyperface is not the system of records, only computation and customer display is required - Turn OFF the posting toggle, and skip this section.
Configure Posting Eligibility if required - this allows you to set additional account-level conditions that must be met before a reward is posted (e.g. account status must be active). This section can be skipped if no additional posting eligibility filter is needed.
Select the Offer Posting method:
Post Immediately - Reward is posted as soon as the offer subscription period ends and the transaction is validated as settled.
Schedule - Reward is posted after a defined number of days from the reference event (typically subscription end). This allows time for transaction reversals and refunds to be accounted for.
If Schedule is selected, configure:
Posting Happens - Number of days to wait before posting the reward.
Days After The - Reference event from which the delay is calculated (e.g. subscription ends).
Enter the Posting Narration (a ledger comment). This is a mandatory system field - enter a relevant description or placeholder text.
Enter the Reversal Narration - a comment for ledger entries when an offer is reversed.
Verify the posting setup and click Save & Continue.

Step 8: Review and Publish
The final step is the Review page. A consolidated summary of all configurations made across the previous steps is displayed here for final verification before the offer is published.

The Review screen will display a summary of all configurations.
Review each section carefully:
Offer Details - Verify offer name, type, outcome, duration, issuer, and program.
User Base - Confirm the enrollment method, Smart Tag cohort filters (if any).
Transaction Eligibility - Verify all condition groups and AND/OR logic.
Offer Calculation - Confirm reward type and calculation values.
Budget - Confirm total allocated budget and thresholds.
Usage Limits - Confirm per-customer earning limits.
Posting Configuration - Confirm posting method, delay days, and narration.
If any section needs correction, navigate back using the left navigation panel.
Once all configurations are verified, click Publish.
The offer will be set to Scheduled status and will automatically transition to Active when the configured Start Date and time is reached.

Note:
Publishing an offer makes it live in the system. Verify all configurations thoroughly before clicking Publish.
Once an offer is Active, only the end date and display settings can be edited. All eligibility, calculation, and posting configurations are locked.
If the offer needs to be stopped before the end date, use the Expire action from the Offer Listing page.
